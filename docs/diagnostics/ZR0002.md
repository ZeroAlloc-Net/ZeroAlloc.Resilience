# ZR0002 — Timeout Configured but No CancellationToken

**Severity:** Warning

A timeout is configured (`[Timeout(Ms = ...)]` or `PerAttemptTimeoutMs > 0` on `[Retry]`) but the method has no `CancellationToken` parameter. The generated `CancellationTokenSource` cannot propagate cancellation to the inner call — the timeout will not actually cancel an in-flight operation.

---

## Examples

### Total timeout without CancellationToken

```csharp
// ZR0002: FetchAsync has no CancellationToken — [Timeout] cannot cancel it
[Timeout(Ms = 5_000)]
public interface IExternalService
{
    ValueTask<string> FetchAsync(string id); // ← no CancellationToken
}
```

### Per-attempt timeout without CancellationToken

```csharp
// ZR0002: PerAttemptTimeoutMs is set but no CancellationToken to pass
[Retry(MaxAttempts = 3, PerAttemptTimeoutMs = 1_000)]
public interface IExternalService
{
    ValueTask<string> FetchAsync(string id); // ← no CancellationToken
}
```

---

## How to fix

Add a `CancellationToken` parameter to the method:

```csharp
[Timeout(Ms = 5_000)]
public interface IExternalService
{
    ValueTask<string> FetchAsync(string id, CancellationToken ct); // ✓
}
```

The generator will pass the timeout-linked token as `ct` to the inner call.

---

## When to suppress

If the inner implementation genuinely cannot be cancelled (e.g. a third-party library with no `CancellationToken` support), suppress the warning with:

```csharp
#pragma warning disable ZR0002
[Timeout(Ms = 5_000)]
ValueTask<string> FetchAsync(string id);
#pragma warning restore ZR0002
```

The timeout CTS will still fire, but the inner call will not observe it — the call will complete normally and the result will be discarded if the deadline has already passed.
